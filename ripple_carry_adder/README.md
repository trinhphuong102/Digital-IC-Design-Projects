# Parameterized N-bit Ripple Carry Adder (RCA)

This repository contains the RTL design, verification, and synthesis of a parameterized N-bit Ripple Carry Adder using Verilog HDL. The project demonstrates a complete ASIC/FPGA front-end flow, from structural RTL coding and randomized testbench verification to logic synthesis and timing analysis using Intel Quartus Prime.

## 1. Architecture Overview
The design utilizes a hierarchical approach to build an N-bit adder.
*   **Structural Modeling:** The top-level module (`seq_tt`) instantiates a chain of Full Adders, which are in turn built from Half Adders. 
*   **Procedural Logic:** Basic combinatorial blocks (like the Half Adder) are implemented using procedural `always` blocks to demonstrate flexible logic modeling.
*   **Scalability:** Advanced `generate` loops and `parameter` definitions are utilized to allow the data width ($N$) to be easily reconfigured without modifying the core logic.

## 2. RTL Synthesis & Schematic
The structural design translates cleanly into hardware. Below is the RTL Viewer schematic generated by Intel Quartus Prime for a **4-bit configuration** ($N=4$). 

![RTL Schematic](schematic.png)

*   **Observation:** The schematic clearly illustrates the "ripple" carry chain. The carry-out (`cout`) of `fa1` (bit 0) connects directly to the carry-in (`cin`) of `gen_adder[1].fa`, continuing up to the most significant bit.

## 3. Timing Analysis (Critical Path)
A core characteristic of the Ripple Carry Adder is that the propagation delay increases linearly with the number of bits. The synthesis report below captures the timing delays for the 4-bit implementation.

![Propagation Delay](datasheet_report.png)

*   **Critical Path Identification:** Based on the Datasheet Report, the longest delay path occurs from the least significant input `b[0]` to the most significant sum bit `s[3]` (up to **7.944 ns**) and the final carry-out `c` (up to **7.729 ns**). This confirms the theoretical bottleneck of RCAs, where the final output must wait for the carry to ripple through all previous stages.

## 4. Verification & Simulation
To ensure design correctness, a self-checking testbench was developed. The simulation below showcases a **32-bit configuration** ($N=32$) subjected to randomized stimuli.

![Waveform](waveform.png)

*   **Methodology:** The testbench generates random 32-bit vectors for inputs `a` and `b`.
*   **Data Representation:** Values are displayed in Hexadecimal (`%8h`) format to efficiently monitor the 32-bit data bus.
*   **Result:** The waveform validates that the sum (`s`) and carry-out (`c`) evaluate correctly across transitions, perfectly matching the expected hardware behavior.
